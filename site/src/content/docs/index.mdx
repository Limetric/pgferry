## What pgferry is for

`pgferry` is for one-shot database migrations into PostgreSQL when you want explicit behavior instead of a black box: schema introspection, predictable table creation, `COPY`-based loading, resumable chunking, hook phases, and post-load validation.

<div class="config-stage">
	<div class="config-panel">

```toml
schema = "app"
on_schema_exists = "error"
unlogged_tables = false
resume = true
validation = "row_count"

[source]
type = "mysql"
dsn = "root:root@tcp(127.0.0.1:3306)/source_db"

[target]
dsn = "postgres://postgres:postgres@127.0.0.1:5432/target_db?sslmode=disable"
```

	</div>
	<div class="config-notes">
		<article>
			<h3>Safe by default</h3>
			<p><code>on_schema_exists = "error"</code> avoids destroying prior target state during rehearsals.</p>
		</article>
		<article>
			<h3>Crash-safe runs</h3>
			<p><code>unlogged_tables = false</code> plus <code>resume = true</code> is the correct long-run pairing.</p>
		</article>
		<article>
			<h3>Sanity check built in</h3>
			<p><code>validation = "row_count"</code> gives you a quick first-pass verification before cutover.</p>
		</article>
	</div>
</div>

## Choose your migration path

<CardGrid>
	<Card title="Minimal-safe" icon="shield">
		Use this first for production rehearsals. Keeps logged tables, avoids destructive schema recreation, and biases toward inspection over speed.
	</Card>
	<Card title="Recreate-fast" icon="bolt">
		Use this for disposable dev or staging targets. Drops the schema, uses <code>UNLOGGED</code> tables, and optimizes for fast repeat runs.
	</Card>
	<Card title="Chunked-resume" icon="refresh">
		Use this for large tables or long-running loads where interruption cost matters more than raw bulk-load speed.
	</Card>
	<Card title="Schema-only / data-only" icon="list">
		Use this when you need more control over DDL review, cutover sequencing, or a separate schema approval step.
	</Card>
</CardGrid>

## The migration flow

<div class="flow-grid">
	<article>
		<span>01</span>
		<h3>Plan</h3>
		<p>Inspect the source, learn what needs manual handling, and generate hook scaffolding.</p>
	</article>
	<article>
		<span>02</span>
		<h3>Prepare</h3>
		<p>Pick a migration pattern, choose snapshot mode, and install any required target extensions.</p>
	</article>
	<article>
		<span>03</span>
		<h3>Load</h3>
		<p>Create tables, stream data with `COPY`, and use chunking or resume when the run is long.</p>
	</article>
	<article>
		<span>04</span>
		<h3>Finalize</h3>
		<p>Add keys, indexes, foreign keys, sequences, and optional trigger emulation after the heavy load is done.</p>
	</article>
	<article>
		<span>05</span>
		<h3>Cut over</h3>
		<p>Check row counts, run application-specific follow-up SQL, and verify the target behaves the way production needs.</p>
	</article>
</div>

## What pgferry handles automatically

<div class="do-dont-grid">
	<article>
		<h3>Handled by pgferry</h3>
		<ul>
			<li>source schema introspection</li>
			<li>table creation and PostgreSQL identifier normalization</li>
			<li>`COPY`-based data loading with chunking support</li>
			<li>post-load PK, index, FK, sequence, and optional trigger creation</li>
			<li>checkpointed resume and optional row-count validation</li>
		</ul>
	</article>
	<article>
		<h3>You still need to own</h3>
		<ul>
			<li>views, routines, and source-trigger replacements</li>
			<li>generated-column expression semantics</li>
			<li>application-specific cutover checks</li>
			<li>unsupported index redesign where needed</li>
			<li>decisions about speed versus safety for your environment</li>
		</ul>
	</article>
</div>
